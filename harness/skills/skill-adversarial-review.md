# Skill: Adversarial Code Review (对抗性审查)

## 调用时机
在 PHASE_4 静态审计通过后，由 AI 自动触发。

## 审查角色
你必须**依次扮演**以下三个角色，并对本次变更的代码进行攻击性思考：

### 角色 1: Security Auditor (安全审计员)
- 检查项：
    - 是否存在 SQL 注入、XSS、CSRF 风险？
    - 认证与授权逻辑是否有绕过可能？
    - 敏感数据（密码、Token）是否误记录到日志？
    - 文件上传、路径遍历、反序列化漏洞。
- 对每个发现给出：
    - 风险等级: `CRITICAL / HIGH / MEDIUM / LOW`
    - 攻击向量描述
    - 建议修复方案

### 角色 2: Resilience Tester (韧性测试员)
- 检查项：
    - 是否存在未捕获的异常会导致服务崩溃？
    - 外部调用是否有超时设置？是否有降级逻辑？
    - 数据库连接、线程池资源是否可能泄漏？
    - 是否存在死锁或活锁风险（乐观锁冲突重试）？
- 对每个发现给出：
    - 失效模式
    - 影响范围
    - 建议加固方案

### 角色 3: Business Logic Fuzzer (业务逻辑破坏者)
- 检查项：
    - 尝试传递非法参数：null、空集合、超大数值、负数、重复ID。
    - 尝试并发调用同一接口多次，是否会产生重复订单/重复扣款？
    - 尝试跳过前端校验直接调用后端，是否能绕开业务约束？
- 对每个发现给出：
    - 攻击步骤
    - 预期行为 vs 实际行为
    - 建议修复
